# B2E スクリプト仕様の改善案

zpaq.b2e の実装を通じて、B2E スクリプト言語の改善が必要な点が明らかになりました。

## 1. セクション存在時の未実装ハンドリング

**現状の問題：**
- セクションが存在するだけで対応メニューが有効になる
- セクション内容が空でも有効と判定される
- zpaq のように削除機能のない場合、削除メニューをグレイアウトできない

**改善案：**
- セクション存在 ≠ メニュー有効と分離する
- セクション内容が空または実装不完全な場合、そのメニューをグレイアウト
- または明示的な「未実装マーク」を定義（例：`(unimplemented)` 等）

**実装例：**
```lisp
; セクションは存在するがメニューは無効にしたい場合
delete:
 (unimplemented)
```

## 2. B2E スクリプトへのコメント機能追加

**現状の問題：**
- スクリプトにコメントが書けない
- 複雑なロジックの意図がわからない
- zpaq.b2e でも出力フォーマットの詳細を注記したい

**改善案：**
- 行頭に特定の文字でコメント行を指定
- 候補：`#` または `;`

**検討中の文法：**

```lisp
; シェルスクリプトと同じ ; を使う場合
load:
 ; zpaq64.exe の圧縮レベルは -m0 (fastest) から -m5 (best)
 (name zpaq64.exe)
 (type zpaq Store *Method1 Method2 Method3 Method4 Method5)
```

または

```lisp
# シェルスクリプトの # を使う場合
load:
 # zpaq64.exe の圧縮レベルは -m0 (fastest) から -m5 (best)
 (name zpaq64.exe)
 (type zpaq Store *Method1 Method2 Method3 Method4 Method5)
```

**推奨：** `;` — Lisp 方言に従い、既存ユーザーが直感的に理解しやすい

## 3. scan/xscan のロジック改善

**現状の問題：**
- `scan A B C D E command...` の仕様では、終了マーカー「C で始まる行」が必須
- zpaq のように終了マーカーが「2.049443 MB...」（数値で始まる）場合、ファイルリストに含まれてしまう
- 多くのアーカイバツール出力フォーマットが異なるため、パース失敗が多い

**zpaq での具体例：**
```
- 2026-06-07 21:48:44       444416 A     AileEx.exe     ← ここまで取得したい
...
- 2016-08-17 21:46:59      1125376 A     bin/zpaq64.exe
2.049443 MB of 2.049443 MB (16 files) shown             ← この行も取得されてしまう
  -> 2.049443 MB (63 refs...)                           ← 終了マーカー行
```

**改善案（複数パターン）：**

### 案1: 終了マーカーを複数指定可能に
```lisp
(scan "^-" 0 ("shown" | "^\\s+->" ) 1 41 l (arc))
```
複数の終了マーカーを OR 条件で指定

### 案2: 空行をマーカーにできるようにする
```lisp
(scan "^-" 0 (blank-line) 1 41 l (arc))
```
ファイルリスト後の空行で終了判定

### 案3: パターンマッチング強化
```lisp
(scan-regex "^-\\s" 0 "^\\s+->" 1 41 l (arc))
```
正規表現をサポート

### 案4: 「データ行の最後」を判定する構文
```lisp
(scan "-" 0 (until-not-matching "^\\s*-") 1 41 l (arc))
```
ファイル行ではじまらない行で終了

**推奨：** 案4 または案3 — 汎用性が高く、様々なアーカイバ出力に対応可能

## 参考：zpaq.b2e での課題

```lisp
list:
 (scan "zpaq v7.15 journaling archiver" 3 "  ->" 1 41 l (arc))
```

このスクリプトは概ね動作していますが、「2.049443 MB...」の行も取得されているため、
不要な統計行がリストに表示される問題を抱えています。

改善されれば、このような課題は完全に解決できます。

## 実装の優先度

1. **低** — コメント機能（利便性向上、既存機能への影響なし）
2. **中** — セクション未実装ハンドリング（UX改善）
3. **高** — scan/xscan 改善（多くのフォーマット対応に必須）
